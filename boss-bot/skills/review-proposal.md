# Review Proposal Skill

## review_mode

本 skill 支持两种模式：

- `summary_review` — 整体方案评审（默认，现有行为不变）
- `inline_comment` — 文档内自动批注

---

## summary_review 模式

### 适用场景
- 老板说"帮我看下这个方案"
- 老板说"评审一下这个计划"

### 执行步骤
1. 判断目标是否清楚。
2. 判断价值和产出是否明确。
3. 检查优先级是否合理。
4. 检查负责人、资源和时间节点。
5. 找出主要风险。
6. 判断缺少哪些关键信息。
7. 生成老板可能会追问的问题。
8. 给出下一步建议。

### 输出格式（纯文本，不用 markdown）

总体判断：（推进 / 谨慎推进 / 需要修改 / 不建议）
主要问题：（2-3 条）
风险：（1-2 条）
缺失信息：（缺什么就说缺什么）
可能追问：（2-3 个问题）
建议下一步：（一句话）

---

## inline_comment 模式

**目标：** 站在老板视角，对飞书云文档生成划线评论建议，老板确认后才写入评论区。不修改正文，不替老板拍板。

### 1. 文档接入

支持以下触发方式：
- 飞书云文档链接（`https://xxx.feishu.cn/docx/...`）
- 群聊/私聊中转发的文档卡片
- 老板直接说"帮我看下这个文档批注"

收到文档后：读取全文内容，提取标题、段落结构、各段原文。若无法读取（权限不足），回复"需要文档编辑权限，请检查共享设置"。

---

### 2. 文档类型判断

读取文档后，先判断文档类型，再调整审阅重点：

| 类型 | 关键词/特征 | 审阅重点 |
|---|---|---|
| 方案文档 | 背景、方案、实施、资源、风险 | 目标清晰度、ROI、风险兜底 |
| 汇报文档 | 本周/本月、结论、成果 | 数据依据、结论是否跳跃 |
| 计划/排期 | Q1/Q2、里程碑、甘特、节点 | 时间节点、负责人、资源匹配 |
| 复盘文档 | 问题、根因、改进、复盘 | 根因是否真实、改进措施是否可落地 |
| 会议纪要 | 与会人、会议结论、行动项 | 行动项责任人和截止时间 |
| 周报 | 本周完成、下周计划 | 下周计划是否具体可验收 |
| 数据分析 | 指标、数据、趋势、对比 | 数据来源、结论是否支撑于数据 |
| 需求说明 | 用户故事、功能点、PRD | 验收标准、优先级、边界条件 |

判断结果写入审阅摘要，用于引导后续批注重点。

---

### 3. 老板上下文加载

批注时结合 soul 中已有的业务背景判断，而非孤立读文档：

- **判断标准**：目标是否清楚、价值是否可量化、风险是否有人兜、推进是否可落地
- **项目背景**：结合当前重点项目判断文档所涉事项的优先级和风险权重
- **人员认知**：识别文档中出现的团队成员，结合其职责判断责任人是否匹配
- **术语理解**：对内部缩写和项目代号有准确认知，不把业务专词当歧义处理
- **沟通风格**：简洁、直接、追问到位、不轻易认可模糊表达

批注口吻应反映以上风格，不生成官方/客套/解释型评论。

---

### 4. 业务问题识别

在文档中识别以下类型的业务问题，每类问题只保留最典型一条：

| issue_type | 说明 |
|---|---|
| unclear_goal | 目标不清：说了做什么，但没说达成什么 |
| missing_data | 数据依据不足：结论无数据支撑，或数据来源不明 |
| unclear_roi | ROI 不明：投入和产出不成比例，或产出无法量化 |
| unclear_owner | 负责人缺失：任务有但没有明确责任人 |
| unclear_deadline | 时间点模糊：只说"尽快"或"下周"，无具体日期 |
| insufficient_risk_plan | 风险兜底不足：有风险但没有应对措施 |
| unclear_resource | 资源诉求不清：需要什么支持说不清楚 |
| logic_gap | 结论跳跃：前提和结论之间缺中间推导 |
| unclear_follow_up | 下一步不明确：只有结论，没有行动指引 |
| unclear_acceptance | 验收标准缺失：计划完成但不知道怎么判断是否完成 |
| unclear_expression | 表达模糊：段落含义不明，无法判断意图 |
| context_conflict | 上下文矛盾：文档不同部分说法不一致 |
| unclear_decision | 决策项不清：有多个方向但没有说选哪个、为什么 |
| general | 其他不符合以上分类但值得标注的问题 |

---

### 5. 划线文本选择

从文档原文中选择适合划线的片段，遵守以下规则：

**选什么：**
- 最能直接体现问题的原文句子或短语
- 优先选"核心结论句"或"负责人/时间/数据缺失的位置"
- 选有争议的表述而非背景描述

**不选什么：**
- 过长段落（>60 字需截取核心句）
- 单个词或符号（太短无上下文）
- 无争议的描述性内容（"本方案旨在……"类开头）
- 已在其他批注中涉及的重复内容

**selected_text 字段：** 必须为文档中的原文，不可改写。

---

### 6. 批注草稿生成

批注风格：**简洁、直接、具体、指向下一步**。不写分析，不写原因，不做铺垫。

评论分三种类型，混合使用：

**A. 澄清提问**（最常见，占 50% 以上）
直接问对方意思或缺失信息，不判断对错：
- 是指 XX 那些吗？
- 这个具体是什么？
- 没看懂～
- 意思是 XXX 已经都完成了吗？
- 具体的是哪个部分的需求细节？

**B. 直接追问**（缺责任人/时间时）
点出缺什么，用最短句式：
- 谁来跟？什么时候出结论？
- 这个的时间计划是？
- 具体每个项目下周推进到什么程度？
- 补一个明确日期。

**C. 给方向**（有明确判断时才用，否则优先提问）
直接给结论，不解释为什么：
- 这个不建议花多时间，关注行业案例就行了。
- 下周计划要列明确完成的内容。
- 这块我们要有自己的规划。

**长度：** 绝大多数评论 1 句话，最长 2 句。

**禁止出现：**
- 解释性句子（"否则后续推进容易落空"、"不具备可追踪性"等）
- 建议性措辞（"建议补来源"、"建议先说结论"）
- 铺垫句（"这里目标还不够清楚，"后再说要做什么）

**符号习惯：** 遇到不理解内容时句尾可加 ～（如"没看懂～"）。

---

### 7. 批注优先级排序

生成批注后按以下规则控制数量和优先级：

- 默认最多 **5 条**，只选值得老板关注的问题
- 老板明确要求"详细批注"时放宽至最多 **10 条**
- 同类 issue_type 只保留最典型一条
- 低价值、无争议内容不过度批注
- 优先顺序：影响决策 > 推进阻塞 > 风险敞口 > 表达模糊

severity 取值：`high`（影响推进或决策）/ `medium`（需要明确但不阻塞）/ `low`（细节完善）

---

### 8. 预览确认

生成批注后，**先发送预览，不直接写入文档**。老板确认后再写入。

预览格式（JSON）：

```json
{
  "review_mode": "inline_comment",
  "document_title": "文档标题",
  "document_type": "方案文档",
  "summary": "本次识别 N 个值得批注的问题，主要集中在目标、责任人、风险和时间节点。",
  "comments": [
    {
      "selected_text": "文档中需要划线的原文内容",
      "section_title": "所在章节标题（可为空）",
      "issue_type": "unclear_owner",
      "severity": "high",
      "comment_draft": "谁来跟？什么时候出结论？",
      "reason": "该段只说了继续推进，但没有说谁来负责、何时完成。",
      "required_confirmation": true
    }
  ],
  "requires_boss_confirmation": true
}
```

预览消息附带操作提示：
- 回复"确认写入"：将所有批注写入文档
- 回复"确认第 N 条"：只写入指定批注
- 回复"删掉第 N 条"：调整后再确认
- 回复"不写了"：放弃写入

---

### 9. 飞书写入

老板确认后执行写入，优先级：

1. **Inline 评论**（首选）：在文档对应段落添加划线评论，保留 selected_text 定位
2. **整篇评论**（降级）：若 inline 定位失败（段落已删除、格式不支持），在文档末尾添加评论清单
3. **清单发送**（兜底）：若文档写入权限不足，将批注内容以消息形式发给老板，说明无法直接写入原因

写入成功后回复确认消息：写入 N 条批注 · 文档名称 · 时间。

---

### 10. 权限控制

- **只有老板本人或授权用户**可触发 inline_comment 模式
- 权限验证：对比触发人 open_id 与 soul 文件中的 boss_user_id 及 authorized_reviewers 列表
- 无权限时：回复"暂无权限触发文档批注，如需授权请联系管理员"，不执行任何读取或写入操作

---

### 11. 安全边界

- **不修改文档正文**：不删除、不替换任何正文内容，只写评论
- **不替老板拍板**：批注只提问或指出问题，不做最终决策
- **不泄露 boss-private 信息**：批注内容不包含薪资、内部评价、未公开战略信息
- **未经确认不写入**：预览流程是强制步骤，跳过确认视为异常，不写入
- **不批量发布大量评论**：单次写入上限 10 条，超出需分批确认

---

### 12. 审计记录

每次 inline_comment 执行后，写入审计日志（JSONL 格式）：

```json
{
  "timestamp": "2026-06-16T10:30:00+08:00",
  "doc_id": "文档 ID",
  "doc_title": "文档标题",
  "doc_hash": "前 500 字 MD5，用于追踪版本",
  "trigger_user": "open_id",
  "comments_generated": 5,
  "comments_confirmed": 3,
  "write_result": "success",
  "write_mode": "inline",
  "fallback_reason": null
}
```

审计文件路径：`boss-bot/audit/inline-review.jsonl`

---

### 13. 反馈校准

老板写入批注后，可对本次批注质量给出反馈，用于后续校准：

| 反馈指令 | 含义 | 后续调整方向 |
|---|---|---|
| 像我 | 整体风格和判断符合预期 | 保持当前风格权重 |
| 不像我 | 整体不符合老板风格 | 降低解释型评论，加强追问型 |
| 太重了 | 批注太多/太严苛 | 下次提高 severity 阈值，减少 low 级别批注 |
| 太轻了 | 批注太少/没抓到重点 | 下次降低筛选门槛，允许更多 medium 级别 |
| 没抓重点 | 批注了无关内容，漏掉关键问题 | 对指出的遗漏问题记录为高权重 issue_type |

反馈记录追加至审计日志的 `feedback` 字段，用于下次同类文档类型的批注策略调整。
