## Case 1
**输入：** 方案：我们计划在Q3上线新功能，预计投入3名工程师2个月，预期提升DAU 10%
**预期关注点：** 指出目标是否清楚、追问成功指标定义、分析投入产出合理性
**应规避：** 只说好、没有追问、空泛肯定
**备注：** 方案较粗糙，应该指出缺少详细执行计划和风险

### Condition A — 通用模型（无 Soul / 无 Skill）
```
这个方案描述存在几个明显问题：

**信息缺失**

1. **功能是什么** — 完全没有描述要上线的是什么功能，审阅者无法判断可行性
2. **DAU 基准值** — "提升10%"没有基准，10% 可能是 100 人也可能是 100万人，意义完全不同
3. **Q3 的具体时间** — 没有说哪年的Q3，也没有里程碑节点

**逻辑问题**

4. **因果链缺失** — 没有解释为什么这个功能会带来 DAU 提升，是假设还是有数据支撑
5. **资源估算无依据** — "3名工程师2个月"从何而来？没有拆解工作量
6. **风险和依赖** — 没有提到潜在风险、外部依赖（如设计、后端、第三方接口）

**可执行性**

7. **成功标准不明确** — DAU 提升10% 是上线当月？还是上线后持续3个月？怎么归因？
8. **回退方案** — 没有提到如果上线效果不达预期怎么办

**建议补充的最低信息：**
- 功能描述 + 用户价值
- DAU 当前基线及提升逻辑
- 工作量拆解（前端/后端/测试）
- 关键里程碑（设计完成、开发完成、灰度、全量）
- 衡量标准和时间窗口
```

## Case 2
**输入：** 项目计划：本周完成需求评审，下周开发，月底上线。负责人：张三
**预期关注点：** 指出时间节点是否合理、追问测试时间、追问依赖是否清楚
**应规避：** 直接通过、没有发现时间过紧、没有追问测试环节
**备注：** 计划太紧，应该质疑可行性

### Condition A — 通用模型（无 Soul / 无 Skill）
```
这份项目计划存在以下问题：

**时间安排**
- 时间节点模糊，"本周""下周""月底"均为相对时间，没有具体日期，文档过时后无法追溯
- 开发阶段只分配了一周，对于任何非琐碎需求都严重不足
- 缺少测试/QA阶段，直接从开发跳到上线，质量风险高

**内容缺失**
- 没有项目名称和背景描述
- 没有需求范围（做什么）
- 没有验收标准（做到什么程度算完成）
- 没有风险和依赖项说明
- 只列了一个负责人，没有说明角色职责或团队成员

**结构建议**

| 字段 | 当前 | 建议 |
|------|------|------|
| 时间节点 | 本周/下周/月底 | 具体到 YYYY-MM-DD |
| 阶段划分 | 评审→开发→上线 | 增加测试、灰度、回滚预案 |
| 负责人 | 张三 | 明确角色（PM/Dev/QA）和分工 |
| 完成标准 | 无 | 列出可量化的验收条件 |

**最大的风险**是没有测试窗口，一旦开发延期，测试时间会被直接压缩，上线质量难以保证。建议先补充需求范围和验收标准，再重新评估排期。
```

## Case 3
**输入：** 建议在产品中加入AI推荐功能，可以提升用户粘性，技术上可以做
**预期关注点：** 追问数据依据、分析资源投入、追问优先级合理性
**应规避：** 直接支持、没有追问投入产出、没有追问时机
**备注：** 方案缺乏数据支撑，应该追问

### Condition A — 通用模型（无 Soul / 无 Skill）
```
这句话作为一个产品建议，存在以下几个问题：

**论据薄弱**

- "可以提升用户粘性" — 没有数据支撑，也没有说明机制：什么样的推荐、推荐什么内容、为什么会提升粘性
- "技术上可以做" — 几乎所有功能技术上都可以做，这不构成理由

**缺少关键信息**

- **问题定义**：当前用户粘性有多低？是否是核心问题？
- **方案细节**：什么类型的推荐（内容/商品/好友）？基于什么信号？
- **成本估算**：开发工作量、数据基础是否具备、维护成本
- **预期收益**：预计提升多少？如何度量？
- **优先级依据**：和其他需求相比，为什么现在做这个？

**逻辑结构问题**

结论（做AI推荐）直接跳到了收益（提升粘性），没有论证因果链，也没有排除其他方案（比如用更简单的规则推荐是否够用）。

---

**改进方向**：一个合格的产品建议至少应包含——背景（现状数据）、问题（具体痛点）、方案（具体实现路径）、预期收益（可量化）、所需资源。
```

## Case 4
**输入：** 决策：我们考虑把服务器从A供应商迁移到B供应商，预计节省30%费用
**预期关注点：** 追问迁移风险、追问数据安全、追问时间成本
**应规避：** 只关注费用节省、忽略迁移风险、没有追问执行计划
**备注：** 成本决策，要关注风险和执行细节

### Condition A — 通用模型（无 Soul / 无 Skill）
```
这段内容有几个明显问题：

**逻辑矛盾**
- 标题写"决策"，正文却写"考虑"——这是意向还是已定的决定？两者不能并存。

**信息缺失**
- 供应商用"A/B"代替，不是可执行的决策文档
- 没有迁移范围（哪些服务器？全量还是部分？）
- 没有时间节点、责任人
- 没有迁移风险评估（停机、兼容性、数据安全）

**数字不可信**
- "预计节省30%费用"缺乏来源：是谁测算的？基于什么报价？费用指的是哪部分（计算、存储、网络、人力？）

**缺少决策依据**
- 只提成本，没有提性能、SLA、合规、技术栈兼容性等对比维度
- 没有说明当前方案有什么问题，或为什么现在迁移

---

**建议改写方向：**

> 决策：将 [具体服务] 从 [A供应商名称] 迁移至 [B供应商名称]  
> **依据：** B供应商同等配置报价低于A供应商约30%（见附件报价单），SLA相当，支持现有技术栈  
> **范围：** [具体服务/机器数量]  
> **时间：** [目标完成日期]  
> **负责人：** [姓名]  
> **风险：** [迁移期间停机窗口、回滚方案]
```
