# Review Inline Skill — Eval 对比报告
> 生成时间：2026/6/16 17:46:47　用例数量：6

## 测试目标
对比通用模型（Condition A）和 Soul + review_inline Skill（Condition B）在文档批注任务上的差异，验证细分场景建设的价值。
核心关注三个维度：①输出格式是否为 JSON ②评论是否符合老板口吻（短、追问、不解释）③是否正确控制批注数量

## Case 1：计划文档-时间节点过紧缺负责人（RI-01）
**场景说明：** 计划时间极紧（总共约3周），无任何负责人，风险写「暂无」。核心考察：是否识别 unclear_deadline + unclear_owner，评论是否简短追问而非给建议

### Condition A — 通用模型
```
以下是对该文档的逐条问题批注：

---

**批注一：目标缺乏量化（严重）**

> "提升激活率"

无法衡量成功与否。应明确：
- 当前激活率基线是多少？
- 目标值是多少（如从 X% → Y%）？
- 用什么口径定义"激活"？

---

**批注二：时间线模糊且存在逻辑矛盾（严重）**

> 需求评审本周 → UI设计下周 → 前端开发2周内 → 测试3天 → 上线月底

问题：
1. 所有节点均为相对时间，无具体日期，无法追踪进度；
2. "2周内"的起算点不明——若从现在算，前端开发和UI设计会重叠；
3. "月底"指Q3的哪个月底（7月、8月还是9月）？
4. 需求评审→UI→前端→测试→上线的总时长超过3周，而Q3只剩约11周，整体排期是否合理？

建议：改为具体日期（如 6/23、7/4、7/18……）并明确各阶段起止点。

---

**批注三：资源投入不完整（中等）**

> 前端1人、产品1人

计划中包含 UI设计、测试两个阶段，但资源表里没有：
- **UI/视觉设计师**（谁来出设计稿？）
- **测试工程师**（3天测试由谁执行？）
- **后端**（新用户引导通常涉及埋点/A/B实验/接口，是否需要？）
- **数据分析**（如何验证激活率提升？）

---

**批注四：风险评估流于形式（中等）**

> 暂无明显风险

这类改版项目的常见风险至少包括：
- 时间线紧张，测试只有3天，质量存在隐患；
- 改版可能影响已有用户流程，引发兼容性或体验回退问题；
- 激活率指标无基线，无法判断上线效果；
- 资源缺口（设计、测试）若临时借调可能延期。

建议认真填写风险项并列出应对措施。

---

**批注五：缺少验收标准（轻微）**

每个阶段"完成"的定义不明确——需求评审完成是指评审会通过？还是需求文档定稿？UI完成是指初稿还是终稿？建议每个阶段附上 **交付物和验收条件**。

---

**总结性意见**

这份计划目前更接近一个初稿提纲，还不具备可执行性。建议补充：量化目标、具体日期、完整资源、真实风险和各阶段交付物，才能作为正式排期使用。

## Case 2：方案文档-目标模糊缺ROI（RI-02）
**场景说明：** 目标停留在「提升体验」「更多GMV」，无具体数字；无投入产出比；无竞品参照。考察是否识别 unclear_goal + unclear_roi，评论是否为追问而非分析

### Condition A — 通用模型
```

---

**【背景】**

> 当前用户浏览转化率偏低

**问题**：缺乏基线数据支撑。"偏低"是定性描述，应补充：
- 当前转化率具体数值是多少？
- 行业基准或竞品水平？
- 数据来源和统计周期？

没有数据，立项依据不充分，后期也无法评估推荐系统的实际贡献。

---

**【方案概述】**

> 利用用户历史行为数据训练推荐模型

**问题**：
1. **数据状况未说明**：历史数据量级、覆盖时间范围、数据质量如何？新用户/新商品的冷启动问题如何处理？
2. **合规风险缺失**：使用用户行为数据需明确是否取得用户授权，是否符合隐私保护要求（如个人信息保护法）。

---

**【预期效果】**

> 提升用户粘性和满意度，带来更多GMV增长

**问题**：全部为定性描述，无可量化目标，无法判断项目是否成功。建议补充：
- 推荐模块上线后，转化率预期提升 X%？
- GMV 增长目标是多少？
- 如何归因（推荐系统 vs 其他因素）？

---

**【技术方案】**

> 采用协同过滤算法，投入数据工程师2人，前端1人，预计3个月完成

**问题较多**：

1. **算法选型过简**：协同过滤在数据稀疏、冷启动场景下效果有限，应说明算法选型理由，以及是否考虑过内容过滤、混合方案等。
2. **人力配置不完整**：推荐系统需要后端工程师部署推荐服务、搭建 API，当前方案未提及。
3. **无里程碑计划**：3个月是粗估，应拆解阶段（数据准备、模型训练、联调、灰度上线）。
4. **缺少 A/B 测试方案**：效果如何验证？是否有实验组和对照组设计？
5. **缺少降级策略**：推荐服务故障时，首页/商品页如何兜底展示？

**问题**：步骤过于笼统，建议补充：
- 负责人是谁？
- 需求评审的时间节点？
- 评审通过后的后续里程碑？

---

**总体建议**：

| 维度 | 当前状态 | 建议补充 |
|------|----------|----------|
| 量化指标 | 无 | 转化率基线、GMV 目标 |
| 风险评估 | 无 | 冷启动、数据质量、服务可用性 |
| 合规 | 未提及 | 隐私合规、用户授权 |
| 成本预算 | 未提及 | 人力 + 基础设施费用 |
| 时间计划 | 粗估 | 分阶段里程碑 + 负责人 |

文档目前更像一个意向说明，还不具备启动立项所需的完整信息，建议在需求评审前先补齐上述缺口。

## Case 4：复盘文档-根因分析跳跃（RI-04）
**场景说明：** 根因「系统压力过大」没有说是什么压力、为什么会有，三条改进措施全是模糊动作，责任人是集体。考察是否识别 logic_gap + unclear_owner，追问到具体根因

### Condition A — 通用模型
```
以下是对该复盘文档的逐项批注：

---

## 整体评价

这份复盘文档**流于形式**，缺乏实质内容。复盘的核心价值在于找到真正的根因并防止复发，但此文档在三个最关键的部分（根因、措施、跟进）均未给出可执行的信息。

---

## 具体问题批注

### 标题
> "门店助手**上线**故障复盘"

**批注：** "上线"二字暗示这是一次版本发布引发的故障，但正文未说明事件是否由部署触发。若是上线引起，应明确是哪个版本；若不是，标题用词有误，应改为"运行故障复盘"。

---

### 原因分析
> "系统压力过大导致服务不稳定"

**批注（最严重问题）：** 这是整篇文档最大的缺陷。"系统压力过大"是现象描述，不是根因。缺少：

2. **缺少真实飞书文档样本** — 技术方案文档（架构图+文字混排）、PRD（交互说明）等真实格式未经测试，块元素定位是否可行待验证
3. **问题密度偏低** — 模拟文档每篇 2-3 个典型问题，真实方案文档可能同时存在 8-10 个问题，≤5 条限制下的优先级筛选压力未被测试
4. **缺少写入飞书的端到端测试** — 本次 eval 只验证批注 JSON 生成质量，writeDocComment 的 inline 定位 + 降级写入逻辑需单独集成测试

## 结论

Skill 建设在 review_inline 场景的核心价值体现在三个维度：

**1. 输出格式的结构化**
通用模型自由输出段落式意见，无法直接写入飞书；Skill 强制 JSON 格式，每条批注含 selected_text / issue_type / comment_draft，可直接对接 writeDocComment。

**2. 评论口吻的风格一致性**
通用模型习惯「建议补充…」「需要注意…」等解释型措辞；Skill 约束了老板追问式口吻（「谁来跟？」「时间计划是？」「没看懂～」），禁止铺垫和解释。

**3. 批注数量的克制**
通用模型倾向穷举所有问题；Skill 限制默认 ≤5 条，只选影响决策和推进的问题，避免老板看到大量低价值批注产生抵触。
